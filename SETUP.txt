------------------------------------------------------------
  A1. BACKEND SETUP (run the API server locally)
------------------------------------------------------------

Step 1 — Open a terminal and navigate to the backend folder:

    cd "Usecase 2 - Spending Metrics/backend"

Step 2 — Copy the example environment file and fill in your key:

    Windows (PowerShell):  Copy-Item .env.example .env
    Mac/Linux:             cp .env.example .env

    Open .env in any text editor and set:
      API_KEY=gsk_your_actual_groq_key_here

    Leave everything else as default for local development.

Step 3 — Install all Python dependencies:

    uv sync

    This creates a .venv folder and installs all 53 packages.
    First run takes 2-5 minutes (compiles some packages from source).

Step 4 — Start the backend server:

    uv run uvicorn app:app --reload --port 8000

    You should see:
      INFO:     Uvicorn running on http://0.0.0.0:8000 (Press CTRL+C to quit)

    Test it is alive:
      Open browser → http://localhost:8000/health
      You should see: {"status":"ok"}

    Keep this terminal open. The server runs until you press CTRL+C.


------------------------------------------------------------
  A2. FRONTEND SETUP (run the React UI locally)
------------------------------------------------------------

Step 1 — Open a NEW terminal and navigate to the frontend folder:

    cd "Usecase 2 - Spending Metrics/frontend/corporate-buddy/spending-metrics"

Step 2 — Copy the environment file:

    Windows (PowerShell):  Copy-Item .env.example .env
    Mac/Linux:             cp .env.example .env

    The default VITE_API_URL=/api works for local development.
    No changes needed.

Step 3 — Install Node.js dependencies:

    npm install

    This installs React, Vite, Tailwind, etc. (~200MB in node_modules).
    Takes about 1-2 minutes on first run.

Step 4 — Start the frontend dev server:

    npm run dev

    You should see:
      VITE v6.x.x  ready in xxx ms
      ➜  Local:   http://localhost:3000/

    Open your browser at http://localhost:3000

  A3. USING THE APP LOCALLY

  2. Click on the Spending Tool / Analyse button

================================================================================
  PART B — AWS DEPLOYMENT (EC2 + S3 + Docker)
================================================================================

WHAT YOU NEED
-------------
  1. AWS account                    https://aws.amazon.com
  2. Docker Desktop installed locally  https://www.docker.com/products/docker-desktop/
  3. Docker Hub account (free)      https://hub.docker.com
     OR use AWS ECR (shown as alternative)
  4. A domain name (optional but recommended for HTTPS)
     Recommended registrar: Namecheap, GoDaddy, or Route53
  5. Your Groq API key (same as local setup)

OVERVIEW OF WHAT WE ARE BUILDING
----------------------------------
  Your PC → builds Docker image → pushes to Docker Hub
  AWS EC2 → pulls Docker image → runs backend container
  AWS S3  → stores your source PDF files
  Nginx   → sits in front, serves frontend + proxies /api to container
  Browser → uploads PDF → Nginx → Docker container → returns results

------------------------------------------------------------
  B1. CREATE S3 BUCKET (stores your source PDFs)
------------------------------------------------------------

  1. Go to AWS Console → S3 → Create bucket
  2. Settings:
       Bucket name:          spending-metrics-data   (must be globally unique)
       AWS Region:           ap-south-1  (Mumbai — closest to India)
       Block all public access: ENABLED  (keep private)
       Versioning:           disabled (not needed)
       Encryption:           SSE-S3 (default)
  3. Click Create bucket.

  4. Upload your PDF statements:
       - Open the bucket → Create folder → name it "monthly"
       - Upload your PDF files into the "monthly/" folder

  5. Note your bucket name — you will need it later.


------------------------------------------------------------
  B2. CREATE IAM ROLE FOR EC2 (no hardcoded AWS keys)
------------------------------------------------------------

  1. AWS Console → IAM → Roles → Create role
  2. Trusted entity type: AWS service
  3. Use case: EC2 → Next
  4. Click "Create policy" (opens new tab):
       - Choose JSON editor and paste:

       {
         "Version": "2012-10-17",
         "Statement": [
           {
             "Effect": "Allow",
             "Action": [
               "s3:GetObject",
               "s3:ListBucket",
               "s3:PutObject"
             ],
             "Resource": [
               "arn:aws:s3:::spending-metrics-data",
               "arn:aws:s3:::spending-metrics-data/*"
             ]
           }
         ]
       }

       Replace "spending-metrics-data" with your actual bucket name.
       - Policy name: SpendingMetricsS3Policy
       - Click Create policy.

  5. Back in the Role creation tab, refresh and search for SpendingMetricsS3Policy.
  6. Select it → Next
  7. Role name: SpendingMetricsEC2Role
  8. Click Create role.


------------------------------------------------------------
  B3. LAUNCH EC2 INSTANCE
------------------------------------------------------------

  1. AWS Console → EC2 → Launch Instance

  2. Fill in these settings:
       Name:                  spending-metrics-server
       Application and OS:    Ubuntu Server 24.04 LTS (HVM), SSD  64-bit x86
       Instance type:         t3.medium  (2 vCPU, 4 GB RAM)
                              NOTE: t2.micro is free-tier but only 1GB RAM —
                              the PDF pipeline needs ~2GB. Use t3.medium.
       Key pair:              Create new key pair
                                Name: spending-metrics-key
                                Type: RSA
                                Format: .pem (Mac/Linux) or .ppk (Windows PuTTY)
                              Download and SAVE this file. You cannot re-download it.

  3. Network settings → Edit:
       Create security group named: spending-metrics-sg
       Add rules:
         Type: SSH,    Port: 22,  Source: My IP        (your home IP only)
         Type: HTTP,   Port: 80,  Source: 0.0.0.0/0
         Type: HTTPS,  Port: 443, Source: 0.0.0.0/0

  4. Configure storage:
       Size: 20 GiB, Volume type: gp3

  5. Advanced details → IAM instance profile:
       Select: SpendingMetricsEC2Role   (created in B2)

  6. Click Launch Instance.

  7. Go to EC2 → Instances → wait for State = "running"
     Note the Public IPv4 address (e.g. 13.233.xxx.xxx) — you need this later.


------------------------------------------------------------
  B4. CONNECT TO EC2 VIA SSH
------------------------------------------------------------

  Windows (PowerShell / Windows Terminal):

    # Fix key file permissions (required on Windows)
    icacls "C:\path\to\spending-metrics-key.pem" /inheritance:r
    icacls "C:\path\to\spending-metrics-key.pem" /grant:r "$($env:USERNAME):(R)"

    # Connect
    ssh -i "C:\path\to\spending-metrics-key.pem" ubuntu@YOUR_EC2_PUBLIC_IP

  Mac/Linux (Terminal):

    chmod 400 ~/Downloads/spending-metrics-key.pem
    ssh -i ~/Downloads/spending-metrics-key.pem ubuntu@YOUR_EC2_PUBLIC_IP

  You should see the Ubuntu welcome banner.
  All commands from here run ON the EC2 server (not your local PC).


------------------------------------------------------------
  B5. INSTALL DOCKER AND NGINX ON EC2
------------------------------------------------------------

  Run these commands on the EC2 server (copy-paste the whole block):

    sudo apt update && sudo apt upgrade -y

    # Install Docker
    curl -fsSL https://get.docker.com | sh
    sudo usermod -aG docker ubuntu

    # Install Nginx + Certbot (for HTTPS)
    sudo apt install -y nginx certbot python3-certbot-nginx

    # Apply docker group (so you don't need sudo for docker)
    newgrp docker

    # Verify
    docker --version
    nginx -v


------------------------------------------------------------
  B6. BUILD AND PUSH DOCKER IMAGE (run on YOUR LOCAL PC)
------------------------------------------------------------

  Open a terminal on your local PC (not EC2):

    cd "Usecase 2 - Spending Metrics/backend"

    # Log in to Docker Hub
    docker login
    # Enter your Docker Hub username and password

    # Build the image
    docker build -t YOUR_DOCKERHUB_USERNAME/spending-metrics:latest .

    # Push to Docker Hub
    docker push YOUR_DOCKERHUB_USERNAME/spending-metrics:latest

  Replace YOUR_DOCKERHUB_USERNAME with your actual Docker Hub username.
  This takes 5-10 minutes (uploads ~1.5GB of layers on first push).

  ALTERNATIVE — Build directly on EC2 (no Docker Hub needed):
    # On EC2: clone or scp your backend/ folder, then:
    docker build -t spending-metrics:latest .
    # Skip the push step; use image name "spending-metrics:latest" below.


------------------------------------------------------------
  B7. RUN THE BACKEND CONTAINER ON EC2
------------------------------------------------------------

  Back on the EC2 server:

    docker pull YOUR_DOCKERHUB_USERNAME/spending-metrics:latest

    docker run -d \
      --name spending-api \
      --restart unless-stopped \
      -p 8000:8000 \
      -e MODEL=meta-llama/llama-4-scout-17b-16e-instruct \
      -e BASE_URL=https://api.groq.com/openai/v1 \
      -e API_KEY=gsk_YOUR_ACTUAL_GROQ_KEY \
      -e DATA_SOURCE=s3 \
      -e S3_BUCKET=spending-metrics-data \
      -e S3_PREFIX=monthly/ \
      -e AWS_REGION=ap-south-1 \
      -e ALLOWED_ORIGINS=http://YOUR_EC2_PUBLIC_IP \
      -e FILES_DIR=/tmp/spending-files \
      YOUR_DOCKERHUB_USERNAME/spending-metrics:latest

  Replace:
    gsk_YOUR_ACTUAL_GROQ_KEY     → your Groq API key
    spending-metrics-data        → your S3 bucket name
    YOUR_EC2_PUBLIC_IP           → your EC2 public IP (e.g. 13.233.xxx.xxx)
    YOUR_DOCKERHUB_USERNAME      → your Docker Hub username

  Verify the container is running:

    docker ps
    curl http://localhost:8000/health
    # Expected output: {"status":"ok"}

  View live logs:

    docker logs -f spending-api


------------------------------------------------------------
  B8. CONFIGURE NGINX
------------------------------------------------------------

  On EC2 — first, add the rate limit zone to the main nginx config:

    sudo nano /etc/nginx/nginx.conf

    Inside the http { } block, add this line (before the include lines):
      limit_req_zone $binary_remote_addr zone=api_limit:10m rate=2r/s;

    Save and exit: CTRL+O, ENTER, CTRL+X

  Now copy the site config:

    sudo nano /etc/nginx/sites-available/spending-api

    Paste the content below, replacing YOUR_DOMAIN with your actual domain
    OR with your EC2 public IP if you don't have a domain yet:

    ----------------------------------------------------------------
    server {
        listen 80;
        listen [::]:80;
        server_name YOUR_DOMAIN www.YOUR_DOMAIN;

        root /var/www/spending-portfolio;
        index index.html;

        location / {
            try_files $uri $uri/ /index.html;
            add_header Cache-Control "public, max-age=3600";
        }

        location /assets/ {
            try_files $uri =404;
            add_header Cache-Control "public, max-age=31536000, immutable";
        }

        location /api/ {
            limit_req zone=api_limit burst=5 nodelay;
            limit_req_status 429;
            client_max_body_size 16M;

            proxy_pass http://127.0.0.1:8000/;
            proxy_http_version 1.1;
            proxy_set_header Host              $host;
            proxy_set_header X-Real-IP         $remote_addr;
            proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_read_timeout    300s;
            proxy_send_timeout    300s;
            proxy_connect_timeout 30s;
            proxy_buffering off;
        }

        server_tokens off;
    }
    ----------------------------------------------------------------

    Save and exit: CTRL+O, ENTER, CTRL+X

  Enable the site and create the web root:

    sudo ln -s /etc/nginx/sites-available/spending-api /etc/nginx/sites-enabled/
    sudo rm -f /etc/nginx/sites-enabled/default
    sudo mkdir -p /var/www/spending-portfolio
    sudo nginx -t
    sudo systemctl reload nginx

  Test:
    curl http://YOUR_EC2_PUBLIC_IP
    # You should get a 404 (frontend not deployed yet) — that is fine.

------------------------------------------------------------
  B10. ENABLE HTTPS WITH FREE SSL (requires a domain name)
------------------------------------------------------------

  If you have a domain, first point it to your EC2 IP:
    Go to your domain registrar → DNS settings → Add A record:
      Name: @         Value: YOUR_EC2_PUBLIC_IP
      Name: www       Value: YOUR_EC2_PUBLIC_IP
    DNS changes take 5-30 minutes to propagate.

  On EC2:

    sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

    Follow the prompts:
      - Enter your email
      - Agree to terms: Y
      - Share email with EFF: N (optional)
      - Redirect HTTP to HTTPS: 2 (recommended)

    Certbot automatically updates nginx config with SSL certificates.

    sudo systemctl reload nginx

  Now update your frontend to use HTTPS:

    On local PC — edit .env:
      VITE_API_URL=https://yourdomain.com/api

    Rebuild and redeploy:
      npm run build
      scp -r dist/* -i "spending-metrics-key.pem" \
          ubuntu@YOUR_EC2_PUBLIC_IP:/var/www/spending-portfolio/

    Also update ALLOWED_ORIGINS in the running Docker container:
      docker stop spending-api
      docker rm spending-api
      # Re-run docker run command from B7 with:
      #   -e ALLOWED_ORIGINS=https://yourdomain.com,https://www.yourdomain.com

    Visit https://yourdomain.com — full app with HTTPS.

  SSL certificates auto-renew. Certbot installs a cron job for this.

================================================================================
  PART E — TROUBLESHOOTING
================================================================================

  ISSUE: "No transactions found in this PDF"
  FIX:   - Make sure your Groq API key is correct in .env
         - The PDF must be a real bank/UPI/CC statement (not a screenshot)
         - Check backend logs: docker logs spending-api

  ISSUE: Backend starts but returns 500 error
  FIX:   docker logs spending-api
         Look for "API_KEY" or "RuntimeError" in the output.
         Usually means the Groq key is missing or wrong.

  ISSUE: Frontend shows "Network Error" or cannot connect to API
  FIX:   - Confirm backend is running: curl http://localhost:8000/health
         - Check VITE_API_URL in frontend/.env matches the backend URL
         - Rebuild frontend after changing .env: npm run build

  ISSUE: Docker container crashes on EC2
  FIX:   docker logs spending-api
         If "killed" — the instance ran out of RAM. Upgrade to t3.medium.

  ISSUE: Nginx 502 Bad Gateway
  FIX:   - Check container is running: docker ps
         - Check container is healthy: curl http://localhost:8000/health
         - Restart container: docker restart spending-api

  ISSUE: PDF upload fails with "413 Request Entity Too Large"
  FIX:   Nginx is blocking it. Check client_max_body_size 16M is in nginx.conf.
         Reload nginx: sudo systemctl reload nginx

  ISSUE: SSL certificate error
  FIX:   sudo certbot renew --dry-run
         If it fails, check your domain DNS points to the correct EC2 IP.

  ISSUE: uv sync fails with build errors (local dev)
  FIX:   Make sure Python 3.12 is installed. Run:
           uv python install 3.12
           uv sync --python 3.12

  USEFUL COMMANDS ON EC2
  -----------------------
    docker ps                          # list running containers
    docker stats spending-api          # CPU / RAM usage of container
    docker restart spending-api        # restart without removing
    sudo systemctl status nginx        # nginx status
    sudo nginx -t                      # test nginx config syntax
    sudo systemctl reload nginx        # apply nginx config changes
    df -h                              # disk usage
    free -h                            # RAM usage

------------------------------------------------------------
  F2. LAUNCH EC2 ON FREE TIER
------------------------------------------------------------

  Follow ALL steps in PART B (B1 through B11) with these changes:

  CHANGE 1 — Instance type in Step B3:
    Choose: t2.micro  (instead of t3.medium)
    This is the free-tier eligible instance.

  CHANGE 2 — After SSH-ing into EC2 (Step B4), add a swap file FIRST:

    This gives the 1 GB instance an extra 2 GB of virtual memory
    so the PDF pipeline does not crash under heavy load.

    Run these commands on EC2 right after connecting:

      sudo fallocate -l 2G /swapfile
      sudo chmod 600 /swapfile
      sudo mkswap /swapfile
      sudo swapon /swapfile
      echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
      free -h

    You should now see "Swap: 2.0G" in the output.
    This persists across reboots automatically.

  CHANGE 3 — Docker run command in Step B7:
    Add this flag to limit container memory and prevent OOM kills:

      --memory="900m" \
      --memory-swap="2900m" \

    Full docker run command for t2.micro:

      docker run -d \
        --name spending-api \
        --restart unless-stopped \
        --memory="900m" \
        --memory-swap="2900m" \
        -p 8000:8000 \
        -e MODEL=meta-llama/llama-4-scout-17b-16e-instruct \
        -e BASE_URL=https://api.groq.com/openai/v1 \
        -e API_KEY=gsk_YOUR_ACTUAL_GROQ_KEY \
        -e DATA_SOURCE=s3 \

  F4. KEEP AWS COSTS ZERO — TIPS
------------------------------------------------------------

  1. Set a billing alert so you get emailed if charges exceed $1:
       AWS Console → Billing → Budgets → Create Budget
       Type: Cost budget → Amount: $1 → Email alert at 80%

  2. Stop EC2 when not using it:
       AWS Console → EC2 → select instance → Instance State → Stop
       Stopped instances do NOT use free-tier hours.
       NOTE: Stopping changes your public IP. Use an Elastic IP
       (free when attached to a running instance) to keep a fixed IP:
         EC2 → Elastic IPs → Allocate → Associate to your instance

  3. Stay within S3 free limits:
       5 GB storage — more than enough for PDF statements
       20,000 GET requests/month — fine for personal use

  4. Monitor free tier usage:
       AWS Console → Billing → Free Tier
       Shows % used for each service this month
